# Tasks

Adjusted from the RND/professor validation on 2026-05-11. This wave is ordered so downstream confidence features do not ship before the evidence substrate exists. The blunt version: fix the Rust semantic path first, then build proof, then build trust, then benchmark the product like it owes us money.

## Runtime Authority And Semantic Correctness (2 tasks)

| # | Task |
|---|------|
| 1 | **Fix Rust Candle/Jina semantic embedder path** - Repair the Rust default embedding backend so `jina-embeddings-v2-base-code` loads and indexes on a fresh target without hashing fallback and without the observed tensor-name/layout failure (`encoder.layer.0.mlp.gated_layers.weight` vs actual Jina safetensor keys such as `up_gated_layer`). This matters because every retrieval benchmark is suspect until the intended semantic backend works. Key behaviors: default `cargo run -p loom-mcp -- reindex --target <fresh-corepack-clone>` must succeed with Candle/Jina; hashing fallback must only run when configured; status must expose backend/degraded/model/dimensions; index fingerprints must invalidate stale or mixed embedding spaces; tests must cover strict failure, configured fallback, stale-vector invalidation, and a network-gated live semantic smoke test. Architectural intent: make Rust the production runtime, not a fast shell around degraded vectors. Boundaries: do not choose a new embedding model unless the current Jina model is proven incompatible after investigation and documented evidence. |
| 2 | **Rust authority and migration cleanup** - Declare the Rust runtime as the only authoritative active implementation and remove stale Python runtime assumptions from docs, benchmark scripts, configs, and agent manuals. This matters because active benchmark machinery still references `uv run python -m loom` and old `.loom.db` paths, which makes regression evidence bad before the run starts. Key behaviors: active benchmark configs must invoke Rust `loom-mcp`; active scripts must use `.loom/loom.db`; historical Python/Rust comparison docs must be archived or labeled as historical research; a guard must fail if active benchmark configs reference `python -m loom`; docs must define Rust tool JSON, status fields, scoring semantics, storage path, schema/version, and benchmark metric contracts. Architectural intent: one product, one runtime, fewer haunted baselines. Boundaries: do not preserve retired runtime compatibility unless explicitly marked historical. |

## Search Proof Contract Foundation (7 tasks)

| # | Task |
|---|------|
| 3 | **Search contract golden fixtures** - Add tiny deterministic in-repo fixtures and tests that prove the beyond-grep contract before and during search-output changes. This matters because "better retrieval" without a fixed contract becomes leaderboard fan fiction with structs. Key behaviors: fixtures must include at least one exact lexical hit in `exact_hits`, one semantic or graph-only result in `beyond_grep`, one duplicate candidate removed deterministically, and one query with empty lexical results that still returns semantic/graph candidates; tests must assert bucket membership, ordering stability, reason codes, file/line anchors, and absence of duplicate file/line spans. Architectural intent: lock the product promise as executable behavior before reshaping tool output. Boundaries: this task may add test fixtures, helper builders, and expected-output snapshots, but it does not by itself implement the production split. |
| 4 | **FTS evidence substrate for exact hits** - Extend the lexical retrieval layer so exact-hit candidates carry evidence, not just symbols. This matters because current FTS returns `Symbol` rows while the contract requires matched text, line anchors, matched field or best-effort field reason, rank, and match reason. Key behaviors: lexical results must expose bounded snippet or matched text, rank, field/reason (`name`, `context`, `file`, `kind`, or best effort), exact phrase vs token-match classification where possible, deterministic punctuation sanitization, and stable result ordering; tests must cover quoted strings, punctuation-heavy identifiers, field matches, empty sanitized queries, and ranking. Architectural intent: make `exact_hits` real evidence instead of a renamed FTS list. Boundaries: this task does not create `evidence_pack`; it may use indexed content or bounded lookup, but must not read unrelated large files wholesale. |
| 5 | **Versioned MCP response contracts and handle compatibility** - Introduce explicit versioned response structs before changing search-family behavior. This matters because silent JSON drift is how agents learn to distrust tools, and frankly they already have enough trust issues. Key behaviors: search, related, impact, neighborhood, inspect, and evidence-pack responses must use named structs rather than ad hoc serialized internals; responses must include a contract/version field, stable handle format tied to index revision, truncation/inspect-required metadata, and compatibility tests for JSON shape; old breaking changes must be deliberate and tested. Architectural intent: make output evolution deliberate, testable, and client-safe. Boundaries: this task does not preserve every old field forever, but any breaking change must be explicit in tests and docs. |
| 6 | **Internal lexical baseline plus beyond-grep split** - Add a Loom-owned exact lexical retrieval stage that runs before semantic/graph expansion and returns explicit buckets: `exact_hits` for Loom lexical results and `beyond_grep` for semantically or structurally relevant symbols/files after excluding exact-hit ids and duplicate file/line spans. This matters because the product promise must be visible: "here is what exact lexical search found, and here is what Loom found beyond that." Key behaviors: exact hits must preserve file, line, matched text/snippet, and match reason; beyond-grep results must include why they survived exclusion; duplicates across buckets must be removed deterministically; empty exact results must still produce semantic/graph results. Architectural intent: lexical search becomes a first-class signal inside Loom, not an external shell fallback. Boundaries: do not claim full grep equivalence unless file-text indexing or bounded file scanning actually covers whole files. |
| 7 | **Compact result handles and inspect workflow** - Change search-family responses to return compact handles, rankings, file/line anchors, one-line summaries, reason codes, and budget/truncation metadata by default, then add an `inspect` tool that resolves handles into bounded source snippets only when requested. This matters because current full symbol contexts burn tokens without giving enough final proof. Key behaviors: handles must be stable within an index revision; stale handles must return actionable errors; `inspect` must support symbol/file handles, line or character budgets, pagination/refusal for large snippets, and citable exact file/line anchors. Architectural intent: make `search` the scout and `inspect` the controlled reader. Boundaries: this task does not add new indexing signals; it changes response shape and follow-up retrieval ergonomics. |
| 8 | **MCP tool descriptions, schemas, and containment policy** - Rewrite the MCP surface so tool descriptions teach the intended sequence: `search` or beyond-grep first, `inspect` only selected handles, `evidence_pack` before final answer, shell only as last resort. Add structured output schemas where supported and mark read-only behavior through MCP annotations/resource links where available. This matters because agent behavior is part of the product, not a charming accident. Key behaviors: descriptions must be compact, explicit, and action-oriented; outputs must separate machine-readable content from display text where possible; all read-only tools must advertise read-only behavior; `reindex` must be explicitly non-read-only; errors must be actionable for retry. Architectural intent: steer Codex away from broad shell reads even when clients ignore annotations. Boundaries: this task does not depend on a particular MCP client honoring annotations; the textual contract must still work. |
| 9 | **Evidence pack tool for final citable proof** - Add an `evidence_pack(query, budget_tokens)` tool only after exact buckets, handles, and `inspect` exist. It must orchestrate lexical hits, beyond-grep results, graph neighbors, role cards when useful, and inspected snippets into a compact final bundle suitable for answering without shell. This matters because the model escapes to `rg`/`nl` when Loom gives candidates but not proof. Key behaviors: the pack must include exact matches, grep-missed findings, source snippets, file/line citations, coverage checklist, omitted/truncated metadata, and missing concepts; it must obey caller-provided token/character budget; it must never read unrelated large files wholesale. Architectural intent: this becomes the primary "stay inside Loom" proof tool. Boundaries: it provides evidence, not the final natural-language answer. |

## Relatable Indexing Signals (4 tasks)

| # | Task |
|---|------|
| 10 | **Behavior fact indexing for constants, env vars, config keys, and CLI strings** - Extend indexing beyond symbols to first-class behavior facts: environment variables, config JSON/YAML/TOML paths, command names, package/script names, error strings, feature flags, and important string literals. This matters because grep wins by seeing operational strings like `COREPACK_ENABLE_STRICT`, while symbol-only search sees only the surrounding function. Key behaviors: facts must link back to file/line spans and nullable enclosing symbols; manifests must produce path-like facts; repeated facts must aggregate occurrences; facts must be searchable lexically and attachable to evidence packs. Architectural intent: make Loom understand the operational surface of a project, not just functions/classes. Boundaries: this does not add full data-flow analysis and does not index secrets beyond local private storage already inside `.loom`. |
| 11 | **Callsite and argument-role indexing** - Index not only that `A` calls `B`, but where the call occurs, the receiver/base object, unresolved target text, resolved target when known, argument literals/summaries, imported aliases, and enclosing symbol. This matters because agents often need "how is this function used?" rather than just "what symbols are related?" Key behaviors: callsite records must include file/line/span, callee, receiver/base, argument summaries, confidence, generic/downweighted flags, and queryable unresolved calls; alias records must be normalized instead of hiding inside import edges. Architectural intent: turn the graph from symbol adjacency into inspectable behavioral paths. Boundaries: no whole-program type inference is required; best-effort static extraction with confidence scores is acceptable. |
| 12 | **File role cards and repository map summaries** - Generate compact deterministic per-file and per-module role cards during indexing: primary responsibility, exported symbols, imported dependencies, behavior facts, centrality, tests touching it, and top related files. This matters because agents need orientation before line-level inspection, and long context is unreliable when relevant facts are buried in the middle. Key behaviors: role cards must be extractive, short, deterministic, incrementally updated, queryable, invalidated by file hash changes, and eligible for evidence packs before snippets when useful. Architectural intent: give the model high-signal orientation without making it read whole files. Boundaries: no LLM summarization dependency is required for the initial implementation. |
| 13 | **Data-flow and state-flow neighborhoods** - Add lightweight data-flow edges after behavior facts and callsites have stable ids. Cover assignments, returns, parameter-to-call propagation, exported values, and config/env value consumption, starting with TypeScript/JavaScript and Rust. This matters because practical agents ask "where does this value come from?" not just "what is nearby?" Key behaviors: data-flow edges must connect variables/parameters/config facts to consuming symbols; returned values must link producer to caller when statically obvious; env/config reads must connect fact ids to functions; low-confidence edges must be visible, not hidden. Architectural intent: make "relatable" mean causal and behavioral, not merely co-located. Boundaries: no full SSA or runtime evaluation. |

## Ranking, Budgets, And Trust (4 tasks)

| # | Task |
|---|------|
| 14 | **Payload budgets and anti-rabbit-hole caps** - Enforce strict per-tool output budgets, default result limits, coupled-result caps, graph expansion diversity caps, pagination, and capping metadata now that handles and `inspect` exist. This matters because more indexed knowledge must reduce exploration, not create more plausible doors to open. Key behaviors: every capped tool must report `truncated`, `next_handle`, `inspect_required`, and omitted counts; broad queries must return summaries and handles, not huge snippets; related/impact/neighborhood must accept limits and diversity caps; callers may explicitly request larger budgets. Architectural intent: detail lives behind explicit inspection. Boundaries: this does not remove advanced detail access. |
| 15 | **Staged hybrid ranking with query intent routing** - Replace single-pass RRF behavior with a staged deterministic pipeline: classify query intent, retrieve lexical/fact/vector/graph/role-card candidates, rerank with intent-aware weights, then cap by evidence diversity and budget. This matters because exact strings, conceptual searches, and impact-analysis questions need different retrieval recipes. Key behaviors: exact symbol/string queries favor lexical and facts; conceptual queries favor vectors and role cards; "what breaks" queries favor dependents/calls/tests; tie-breaks prefer central files and high-confidence edges; diversity prevents ten variants of one file. Architectural intent: retrieval becomes adaptive instead of one-size-fits-all RRF soup. Boundaries: no learned reranker is required if deterministic heuristics are benchmarked. |
| 16 | **Trust signals and coverage accounting** - Add explicit coverage metadata to search and evidence outputs after evidence packs exist: matched concepts, missing concepts, exact-vs-inferred distinction, confidence, inspection-needed state, and why the result is enough to stop searching or why it is not. This matters because the model shells out when it cannot tell whether Loom covered the request. Key behaviors: each result must explain whether it is exact, fact, semantic, graph, data-flow, or co-change; evidence packs must show concept coverage; low-confidence or missing areas must be visible; `stop_condition_satisfied` must be computable from evidence coverage, not vibes in a lab coat. Architectural intent: replace model paranoia with machine-readable uncertainty. Boundaries: this does not guarantee correctness; it makes uncertainty explicit. |
| 17 | **Benchmark-gated ranking and containment regression suite** - Add focused regression tests that prove ranking, budgets, evidence packs, and stop conditions improve the actual north-star behavior before running the final Corepack gate. This matters because local unit tests can pass while the agent still runs back to shell like it heard a noise. Key behaviors: tests must include expected evidence sets, exact-hit usefulness, beyond-grep usefulness, shell-escape causes, payload budget assertions, and final-answer evidence coverage; failures must identify whether the gap is retrieval, evidence, confidence, or model habit. Architectural intent: make product behavior testable before expensive agent runs. Boundaries: this does not replace the final external head-to-head; it prevents obviously bad builds from reaching it. |

## Benchmarking And Product Gates (3 tasks)

| # | Task |
|---|------|
| 18 | **Deterministic benchmark harness for grep, Octocode, and Loom versions** - Turn the ad hoc benchmark machinery into a repeatable harness that runs fresh clones, fixed prompts, isolated MCP configs, event parsing, and comparable reports across grep/no-MCP, Octocode, current Loom, and future variants. This matters because one run can be noisy and seductive, which is also how bad statistics get tenure. Key behaviors: each run must record wall time, MCP calls, shell calls, input/output/reasoning/cached tokens when available, MCP chars, shell chars, setup failures, repo commit, prompt hash, model, sandbox, tool timeouts, index fingerprint, and final-answer quality checklist; runs must be sequential or explicitly declared parallel; artifacts must live under `tmp/`; reports must include table and verdict. Architectural intent: make benchmark evidence cheap enough to run after every retrieval change. Boundaries: this does not optimize retrieval itself. |
| 19 | **Shell-escape and useful-symbol metrics** - Add first-class benchmark metrics for the real north star: shell escape rate, shell output chars, useful symbols discovered per token, beyond-grep useful hits, and evidence completeness. This matters because token totals alone can hide whether faster runtime actually keeps the model inside Loom. Key behaviors: every benchmark task must define required files/functions/facts/relationships, aliases, source spans when known, exact-hit vs beyond-grep usefulness, evidence completeness, and shell-escape attribution (`missing_evidence`, `missing_exact_lines`, `missing_confidence`, `verification_only`, `model_habit`, `setup_failure`); reports must show regressions against baseline. Architectural intent: measure the product behavior we want, not tool-call vanity numbers. Boundaries: start with curated expected evidence sets; do not require perfect human labels for every future task. |
| 20 | **Corepack containment acceptance gate** - Create an executable acceptance benchmark for the package-manager-shim Corepack task that Loom must pass before claiming "beats grep": answer quality equal or better than grep/no-MCP, lower shell output, lower or comparable input tokens, and visible beyond-grep value. This matters because Corepack is the first honest embarrassment-driven benchmark. Key behaviors: the gate must run on fresh Corepack clones at the same commit; expected evidence must include shim generation, `runMain`, `executePackageManagerRequest`, `findProjectSpec`, `parsePackageJSON`, `devEngines`, env vars, transparent commands, defaults, install, and `runVersion`; failure must produce a diagnosis table. Architectural intent: make "beat grep" a measurable release gate. Boundaries: Corepack is the first gate, not the only benchmark forever. |

## Required Execution Order

The wave runner may group adjacent compatible tasks into pipelines, but it must preserve these dependency boundaries. Do not run later groups until earlier groups are complete and verified.

| Order | Tasks | Dependency rule |
|---:|---|---|
| 1 | 1 | Fix live Rust Candle/Jina first. If semantic reindex fails, stop product-claim work. |
| 2 | 2 | Remove stale Python/runtime assumptions before benchmark work uses them again. |
| 3 | 3, 4, 5 | Lock search contracts, lexical evidence, and versioned response/handle compatibility. |
| 4 | 6, 7, 8, 9 | Build exact/beyond buckets, compact handles, MCP containment policy, and evidence packs. |
| 5 | 10, 11, 12 | Add behavior facts, callsites, and role cards. |
| 6 | 13 | Add data/state flow only after facts and callsites have stable ids. |
| 7 | 14, 15, 16, 17 | Add budgets, staged ranking, trust coverage, and regression gates. |
| 8 | 18, 19, 20 | Build deterministic benchmark harness, metrics, and Corepack acceptance gate. |
| 9 - LAST | Final Corepack one-by-one head-to-head measurement run | Run a Corepack task once with grep/no-MCP only, wait for it to finish completely, then run the exact same task once with the new Loom only. Use fresh Corepack clones at the same commit, same model, same prompt except tool availability, isolated configs, and sequential execution only. Measure every available metric: start/end timestamps, wall time, exit status, model, sandbox, prompt hash, repo commit, setup time/failures, index build time, index size, vector backend, embedder backend/status/fingerprint, MCP server startup time, tool-call count, per-tool latency, per-tool request/response chars, MCP calls by tool, shell calls by command, shell output chars, file-read count, input/output/reasoning/cached tokens when available, total event count, final answer chars, cited files/lines, required evidence coverage, exact-hit useful symbols, beyond-grep useful symbols, useful symbols per token, shell escape rate, shell escape cause labels, missing evidence, answer-quality checklist, diff from expected evidence set, and regression verdict. Produce a final report under `tmp/benchmark/corepack-gate/` with tables, artifacts, diagnosis, and a clear verdict: `LOOM_BEATS_GREP`, `LOOM_TIES_GREP`, or `LOOM_FAILS_GATE`. |
