# Wave 2 — Full Rust Rewrite

> "The Python served us well. It proved the thesis. Now we pour it in steel." — The Professor ☕

**Status:** PLANNED
**Date:** 2026-05-11
**Triggered by:** 60GB RAM incident during indexing + RR research confirming Rust stack maturity
**Goal:** Rewrite Loom entirely in Rust — same feature parity, 10-50x better resource efficiency, single-binary distribution.
**Research:** `docs/dev/research/rust-techniques-2026-05-11.md`

---

## Context: Why Rust, Why Now

The Python Loom proved the thesis — vector search + structural coupling beats grep 2.1x for AI code understanding. But the ONNX Runtime's CPU arena allocator ate 60GB of RAM during indexing, and Python's GIL prevents true parallel parsing. The RR research confirmed: **candle eliminates the arena problem entirely** (mmap'd safetensors, ~1.5GB vs 60GB), and Rust's ownership model gives us zero-cost abstractions for the entire pipeline.

The Rust ecosystem is ready: `rmcp` (official MCP SDK), `candle` (HuggingFace ML), `rusqlite` (SQLite), `petgraph` (graphs), `tree-sitter` (parsing), `rayon`+`tokio` (parallelism). Every component has a production-proven Rust crate.

**Key architectural decisions (from RR research):**
- **candle, NOT ort/ONNX** — native Rust ML, mmap'd safetensors, Metal on macOS, no arena allocator
- **rayon for CPU parallelism, tokio for IO** — bridge via oneshot channels, never mix
- **petgraph CSR** — 10-20x less memory than NetworkX for graph operations
- **Cow<str>** for symbol names during parsing — owned String only at persistence boundary
- **Bounded channels everywhere** — unbounded = OOM when embedder lags
- **One Parser per rayon thread** — `parse()` takes `&mut self`, `Language` is `Arc`-shareable

# Tasks

## Foundation Layer (3 tasks)

| # | Task |
|---|------|
| 1 | **Cargo workspace + core types + config** — Initialize the Rust project as a Cargo workspace with `loom-core` (types, config), `loom-mcp` (server binary) crates. Define all core types as Rust structs/enums mirroring Python's `models.py`: `Symbol`, `Edge`, `ParsedEdge`, `CoupledSymbol`, `SearchResult`, `CouplingScore`. Use `serde` derive for serialization. Config via `LoomConfig` struct loaded from `.loom/config.toml` with the same defaults as Python's `config.py` (embedding model name, coupling weights w1/w2/w3, thresholds). Error types via `thiserror` for library errors, `anyhow` at binary boundaries. **Why:** Every other task imports these types — this is the foundation. **Key behaviors:** `cargo build` compiles clean, `cargo test` passes with unit tests for config loading and type serialization. Config falls back to defaults when `.loom/config.toml` is missing. **Architectural intent:** Workspace structure allows future crates (e.g., `loom-cli`) without circular deps. Types use `Cow<'_, str>` where lifetime-borrowing from source text is possible, owned `String` at persistence boundaries. **Boundaries:** No database, no parsing, no server — pure types and config only. |
| 2 | **SQLite store with rusqlite** — Port `store/db.py` to Rust using `rusqlite` with `bundled` feature (zero-dep SQLite distribution). Implement the full schema: `symbols`, `edges` (ID-based with FK constraints), `index_meta`, `cochange` tables. FTS5 virtual table for keyword search. sqlite-vec for vector storage and brute-force cosine similarity. WAL mode enabled by default. Connection architecture: 1 writer behind `Mutex` + N readers via `r2d2` connection pool. All CRUD operations from Python's `LoomDB`: `insert_symbol`, `insert_edge`, `insert_embedding`, `search_fts`, `search_vectors`, `remove_file`, `get_edges_from/to`, `get_unresolved_edges`, `resolve_edge`, plus the 5-strategy fuzzy symbol lookup. Batched transactions for bulk inserts (indexer writes). **Why:** The store is Loom's persistence backbone — every other component reads from or writes to it. **Key behaviors:** Schema auto-creates on first run in `.loom/` directory. `remove_file` cascades deletes via FK constraints (no manual cleanup). Content-hash dedup in `index_meta` for incremental indexing. Vector search returns top-K by cosine similarity. FTS5 search returns ranked results. Concurrent reads don't block each other; writes are serialized. **Architectural intent:** `rusqlite` `bundled` feature compiles SQLite from source — no system dependency. sqlite-vec loaded as extension via `load_extension`. r2d2 pool sized to `num_cpus`. Prepared statements cached for hot-path queries. **Boundaries:** No graph algorithms (that's petgraph), no search fusion (that's the search engine), no embedding generation. Store is pure CRUD + raw search primitives. |
| 3 | **Graph engine with petgraph** — Port `store/graph.py` to Rust using `petgraph` in CSR (Compressed Sparse Row) format. Build the symbol graph from edges in SQLite on startup. Implement: BFS/DFS traversal with configurable depth, `impact_radius` (transitive dependents), `neighborhood` (coupled symbols within N hops), centrality scoring (degree centrality for hub detection). Graph is rebuilt from SQLite edges on startup and updated incrementally when files are re-indexed. **Why:** The graph enables transitive impact analysis — "what breaks if I change this?" requires multi-hop traversal that flat SQL queries can't do efficiently. **Key behaviors:** CSR format uses ~8MB for 1M edges (vs ~200-400MB in NetworkX). BFS to depth 3-4 completes in sub-millisecond. `impact_radius` returns all transitive dependents with depth-decayed confidence scores. Graph handles unresolved edges gracefully (nodes without targets are terminal). Incremental update: when a file is re-indexed, remove its nodes/edges and re-insert from SQLite. **Architectural intent:** petgraph CSR is immutable after construction — rebuild on mutation. For the indexer's hot path this is fine (rebuild is <10ms for 100K edges). Store `NodeIndex` ↔ `symbol_id` mapping in a `HashMap` for O(1) lookup. **Boundaries:** No vector similarity (that's the store), no scoring/fusion (that's the search engine). Graph provides raw traversal primitives only. |

| 4 | **Tree-sitter parsers: JS/TS + Python adapters** — Port the `LanguageAdapter` protocol and adapter registry to Rust as a trait + registry pattern. Implement adapters for JavaScript, TypeScript, and Python using the `tree-sitter` crate with `tree-sitter-javascript`, `tree-sitter-typescript`, `tree-sitter-python` grammar crates. Each adapter extracts: functions, classes, methods, constants, type aliases, interfaces/protocols, and their edges (calls, imports, extends, implements, instantiates, type references). The adapters must extract richer edges than the Python version — full call expressions, receiver types, `this.X` resolution. **Key behaviors:** Parser handles malformed files gracefully (tree-sitter produces partial ASTs). Each adapter returns `Vec<Symbol>` + `Vec<ParsedEdge>`. TypeScript adapter handles both `.ts` and `.tsx`. Python adapter handles decorators, dataclasses, `__init__` methods. Import resolution preserves full paths (relative and absolute). **Architectural intent:** Trait-based dispatch (`dyn LanguageAdapter`) with adapter registry keyed by file extension. Grammar crates are statically linked (~50-200KB each). Thread-local `Parser` instances avoid `&mut self` contention. `bumpalo` arena allocator for per-file AST node allocation — extract owned data, drop arena. **Boundaries:** No Go/Java/Rust/C# — those are Task 5. No embedding, no storage — parser returns data, caller decides what to do with it. |
| 5 | **Tree-sitter parsers: Go, Java, Rust, C# adapters** — Implement language adapters for Go, Java, Rust, and C# using their respective `tree-sitter-*` grammar crates. Same trait interface as Task 4. Go: functions, methods (receiver types), structs, interfaces, goroutine calls. Java: classes, methods, interfaces, annotations, generics, package imports. Rust: functions, structs, enums, traits, impl blocks, `use` statements, macro invocations. C#: classes, methods, interfaces, properties, namespaces, `using` statements. **Why:** Feature parity with the Python version requires all 6+ language families. Go and Java are enterprise staples; Rust and C# round out the coverage. **Key behaviors:** Each adapter handles language-specific edge cases: Go's implicit interfaces, Java's nested classes, Rust's trait implementations and `impl` blocks, C#'s partial classes. All adapters produce the same `Symbol` + `ParsedEdge` types. Adapters register themselves at compile time via the registry. **Architectural intent:** Same trait dispatch as Task 4. Grammar crates add ~2-3MB total binary size for all 15+ grammars. Each adapter is a separate module under `src/parsers/`. **Boundaries:** No new languages beyond Go/Java/Rust/C#. Same parser infrastructure from Task 4 — this task only adds adapter implementations. |

## Indexer Pipeline (3 tasks)

| # | Task |
|---|------|
| 6 | **Candle embedding engine** — Implement the embedding subsystem using `candle-core`, `candle-nn`, `candle-transformers` with `tokenizers` crate. Load `jina-embeddings-v2-base-code` model from safetensors format (mmap'd, zero-copy weight loading). Support Metal backend on macOS (`candle-metal-kernels`), CPU fallback everywhere else. Batch embedding interface: accept `Vec<String>`, return `Vec<Vec<f32>>`. Model downloaded on first run to `~/.loom/models/` (same UX as Ollama). Embedding dimension: 768 floats. **Why:** candle eliminates the ONNX Runtime's 60GB arena allocator problem entirely. mmap'd safetensors means ~1-1.5GB working memory for the 161M parameter model, not 60GB. Native Rust — no C++ FFI, no shared library distribution headaches. **Key behaviors:** First run downloads model (~300-547MB) with progress bar to stderr. Subsequent runs mmap from disk (instant load). Batch sizes tuned per device (Metal can handle larger batches than CPU). Thread-safe for concurrent `encode()` calls — single `Arc<Model>` shared across pipeline. Returns raw `f32` vectors for direct SQLite BLOB storage (768 * 4 = 3072 bytes per vector). Graceful fallback: if Metal unavailable, falls back to CPU with warning. **Architectural intent:** candle with safetensors = mmap'd weights, no deserialization, no arena. JinaBERT (ALiBi positional encoding) is first-class in candle. `tokenizers` crate loads `tokenizer.json` natively. SimSIMD or `std::arch` intrinsics for post-embedding cosine similarity if needed (sqlite-vec already has SIMD internally). **Boundaries:** No quantization in v1 (INT8/GGUF for BERT is underdocumented — prototype separately). No model switching at runtime — single model, configured at startup. |
| 7 | **Indexer pipeline + two-phase resolver** — Implement the full indexing pipeline as a staged channel architecture: `ignore::WalkParallel` (file discovery, gitignore-aware) → bounded mpsc → `rayon` thread pool (one `Parser` per thread, produces `Symbol` + `ParsedEdge`) → bounded mpsc → candle embedder (single `Arc<Model>`, large batches) → bounded mpsc → SQLite writer (`tokio::task::spawn_blocking`, batched transactions). Two-phase edge resolution: Phase 1 inserts all symbols + unresolved edges, Phase 2 queries the full symbol table to resolve `target_id` on edges. Content-hash (SHA-256) dedup via `index_meta` table — only re-parse changed files. **Why:** This is Loom's core engine — file changes become indexed, searchable symbols. The channel architecture provides backpressure (bounded channels prevent OOM when embedder lags), parallelism (rayon for CPU-bound parsing, tokio for IO-bound DB writes), and pipeline efficiency (stages run concurrently). **Key behaviors:** `ignore::WalkParallel` uses work-stealing across `num_cpus` threads, respects `.gitignore` + `.loomignore`. Parser threads produce symbols with `source_id` already assigned (pre-allocated IDs or insert-and-return). Embedder receives batches of symbol texts, returns vectors. Writer batches inserts into transactions of 500-1000 rows. Phase 2 resolution runs after all files parsed — queries unresolved edges, matches `target_name` against symbol table with the 5-strategy fuzzy lookup, sets `target_id` + `confidence`. Full reindex: clear all tables, re-walk. Incremental: compare content hashes, only re-parse changed files, re-run Phase 2 for affected edges. **Architectural intent:** Bounded channels sized at `batch_size * 2` for embedder, `32` for parser output. Never call `par_iter()` on tokio threads (causes priority inversion). Bridge rayon→tokio via `tokio::sync::oneshot`. Budget: `tokio` runtime on `num_cpus / 2` threads, `rayon` pool on `num_cpus` threads. **Boundaries:** No file watching (that's Task 8). No search (that's Task 9). Pipeline is invoked explicitly via `reindex()`. |
| 8 | **File watcher + git analyzer** — Implement file watching using the `notify` crate (v7.x) with debounced events. Same debounce logic as Python's `DebouncedHandler`: collect changed file paths over a configurable window (default 500ms), then trigger incremental re-index for changed files. Git evolutionary coupling analyzer: parse `git log --follow --name-only` output to extract co-change pairs (files that change together in the same commit). Store co-change data in the `cochange` table with frequency + recency scores. **Why:** File watching enables real-time index freshness — edit a file, Loom's index updates within seconds. Git co-change is Signal 3 (evolutionary coupling) — the 15-30% of relationships that structural + semantic signals miss. **Key behaviors:** Watcher ignores `.loom/`, `node_modules/`, `.git/`, and patterns from `.loomignore`. Debounce window is configurable. Watcher triggers incremental pipeline (only changed files). Git analyzer runs on `reindex()` — not on every file change (too expensive). Co-change pairs are stored as `(symbol_a_id, symbol_b_id, frequency, recency)`. Recency decays with commit age. Analyzer handles renamed files via `--follow`. **Architectural intent:** `notify` crate is the same file watcher used by rust-analyzer, Zed, and Deno. Debounce uses `tokio::time::sleep` with coalescing. Git analyzer shells out to `git log` (parsing git output is simpler and more robust than libgit2 for this use case). **Boundaries:** No background daemon mode in v1 — watcher runs while MCP server is running. No real-time git analysis (only on explicit reindex). |

## Search + Server (3 tasks)

| # | Task |
|---|------|
| 9 | **Search engine + coupling scorer** — Port `search/engine.py` and `search/scoring.py` to Rust. Hybrid search: FTS5 keyword search + sqlite-vec vector search, fused via Reciprocal Rank Fusion (RRF, k=60). Coupling scorer computes `coupling(A, B) = w1 * structural(A, B) + w2 * semantic(A, B) + w3 * evolutionary(A, B)` with configurable weights. Search results are expanded with coupled symbols above threshold. Implements all 4 query types: `search(query)` — hybrid FTS+vector with coupled expansion; `related(symbol)` — all coupled symbols with scores + reasons; `impact(symbol)` — blast radius via graph traversal + evolutionary co-changers; `neighborhood(file, line)` — coupling neighborhood of a code location. **Why:** The search engine is what MCP tools call — it's the user-facing intelligence layer. RRF fusion is proven (Cormack et al. 2009) to outperform any single signal. Three-signal coupling scores are what make Loom fundamentally different from grep. **Key behaviors:** `search()` returns top-K results, each expanded with top-3 coupled symbols. `related()` returns all symbols above coupling threshold (configurable, default 0.3). `impact()` uses petgraph BFS for transitive structural dependents + co-change partners. `neighborhood()` finds the symbol at (file, line), returns its coupling neighborhood. All queries return `CoupledSymbol` with score breakdown (structural, semantic, evolutionary) and human-readable reason strings. RRF fusion handles different result set sizes gracefully. **Architectural intent:** Search engine holds `Arc` references to store, graph, embedder. All queries are read-only (no locking beyond r2d2 pool). Vector similarity computed by sqlite-vec (already SIMD-optimized). Coupling computation is pure arithmetic — no allocation in hot path. **Boundaries:** No caching layer in v1. No query parsing/NLP — queries are raw strings passed to FTS5 and embedder. |
| 10 | **MCP server with rmcp + CLI** — Implement the MCP server using `rmcp` (official Anthropic Rust MCP SDK, v1.6.0) with stdio transport. Register 6 tools matching the Python version: `search`, `related`, `impact`, `neighborhood`, `reindex`, `status`. Use `#[tool_handler]` macro for auto-generated tool discovery and JSON schema registration. CLI entry point via `clap`: `loom-mcp` starts the stdio MCP server, `loom-mcp reindex` triggers manual reindex, `loom-mcp status` prints index stats. Server initializes on first tool call: loads config, opens/creates `.loom/` database, builds in-memory graph from edges. **Why:** The MCP server is how AI tools (Claude Code, Cursor, Copilot) consume Loom. stdio transport is what all local MCP clients expect. rmcp is the official SDK — maintained by Anthropic, 3.4K stars, battle-tested. **Key behaviors:** Server starts in <100ms (no model loading until first search/reindex). Tool schemas auto-generated from Rust types via rmcp macros. `reindex` tool triggers full pipeline, streams progress to stderr. `status` returns: total files, total symbols, total edges, index size on disk, last indexed timestamp, stale file count. Error responses follow MCP error format. Server handles concurrent tool calls (multiple AI agents querying simultaneously). **Architectural intent:** rmcp stdio transport reads JSON-RPC from stdin, writes to stdout. All tool handlers are async (tokio). `#[tool_handler]` macro generates `ServerHandler` impl. Lazy initialization: DB + graph + model loaded on first tool call, not on server start. `Arc<SearchEngine>` shared across all tool handlers. **Boundaries:** No SSE/HTTP transport in v1 (stdio only). No authentication (local tool, runs as user). No streaming results (batch response per tool call). |
| 11 | **Cross-platform distribution** — Set up binary distribution using `cargo-dist` for GitHub Releases (macOS aarch64 + x86_64, Linux x86_64 + aarch64, Windows x86_64) and `maturin` for PyPI distribution (`pip install loom-mcp` installs the Rust binary as a platform wheel — same approach as ruff). Homebrew tap via cargo-dist auto-generated formula. CI workflow: GitHub Actions builds + tests on all platforms, cargo-dist generates release artifacts on tag push. `cargo binstall loom-mcp` for Rust users. **Why:** Distribution is how users get Loom. `pip install` is the lowest-friction path for AI tool users (they already have Python). cargo-dist handles the complex matrix of platform builds + code signing. Single binary with bundled SQLite — no runtime dependencies except the embedding model (downloaded on first run). **Key behaviors:** `pip install loom-mcp` → downloads platform-specific wheel → `loom-mcp` binary available on PATH. `brew install loom-org/tap/loom-mcp` on macOS. GitHub Release page has downloadable binaries for all platforms. macOS binary is ad-hoc signed (cargo-dist handles this) to avoid Gatekeeper blocks. Binary size target: ~15-25MB (Rust + bundled SQLite + tree-sitter grammars). Embedding model (~300-547MB) downloaded to `~/.loom/models/` on first `reindex`. **Architectural intent:** Maturin builds the Rust binary and wraps it in a Python wheel with platform tags — no Python code in the wheel, just the binary. cargo-dist generates `.tar.gz` (Unix) and `.zip` (Windows) with install scripts. CI matrix: `{os: [macos-14, ubuntu-22.04, windows-2022], target: [aarch64-apple-darwin, x86_64-apple-darwin, x86_64-unknown-linux-gnu, aarch64-unknown-linux-gnu, x86_64-pc-windows-msvc]}`. **Boundaries:** No musl static linking in v1 (glibc is fine for most Linux). No Windows ARM. No auto-update mechanism. |
